# constitution.md — Ocean Explorer Kata
# Version: 1.4.0

## Purpose
This document defines the non-negotiable principles, assumptions, and
architectural guardrails for the Ocean Explorer Kata. Every planning,
implementation and test decision made by a human or AI agent must
comply with these articles. No exceptions without explicit documented
justification.

---

## Article 1 — Stack (Non-Negotiable)
- Language: Python 3.12
- Framework: FastAPI (latest stable)
- Validation: Pydantic v2 — all input and output models must be typed
- Testing: pytest + FastAPI TestClient (httpx)
- Containerisation: Docker + docker-compose (single command to run: `docker compose up`)
- No additional runtime dependencies without explicit justification in plan.md

## Article 2 — Test-Driven Design (Non-Negotiable)
- Tests are written BEFORE implementation. Always. No exceptions.
- Red → Green → Refactor cycle must be followed for every component
- Three test layers must exist:
  - `tests/unit/` — isolated class/function tests
  - `tests/integration/` — FastAPI endpoint tests via TestClient
  - `tests/acceptance/` — BDD scenario tests derived from bdd-scenarios.md
- A feature is not done until all three layers pass

## Article 3 — Clean Code (Non-Negotiable)
- No function or method longer than 20 lines
- No function with more than 3 parameters (use Pydantic models to bundle)
- Single Responsibility: each class does one thing
- No magic numbers or strings — use enums or constants
- All public methods must have docstrings
- Edge cases must be explicitly considered and tested for every feature
- Design must support modularity and extensibility — domain classes
  must be independently usable and replaceable without cascading changes

## Article 4 — Grid Assumptions (Non-Negotiable)
These were researched and explicitly chosen. They must not be changed
by an AI agent without human approval:
- Origin (0,0) is bottom-left
- North of (x,y) is (x, y+1) — y increases northward
- Default grid size: 10x10, configurable at initialisation
- Boundary behaviour: HARD STOP — probe halts and reports, does NOT wrap
- Obstacle behaviour: probe moves to last valid position and reports obstacle
- Directions: N, S, E, W only — no diagonals
- Turns: 90 degrees only

## Article 5 — Modularity and Extensibility
- Core domain logic (Grid, Probe, Direction) must be independent of FastAPI
- Domain classes must be importable and testable without starting the server
- API layer is a thin wrapper over domain logic only
- Design must support future extension (e.g. multiple probes, 3D grid)
  without requiring rewrites of existing classes

## Article 6 — API Design
- All commands validated by Pydantic at the API boundary
- Meaningful error responses with HTTP status codes:
  - 200: success
  - 400: invalid command or out-of-bounds initialisation
  - 409: move blocked (obstacle or boundary) — probe position returned
- OpenAPI docs auto-generated by FastAPI at /docs — no manual documentation required
- API is stateful within a session (probe state held in memory)

## Article 7 — Non-Functional Requirements
- API must respond within 200ms for any command sequence under 1000 commands
- All endpoints must be covered by integration tests
- Python code must be PEP8 compliant — enforced via ruff with zero warnings
- No hardcoded state — probe and grid initialised via API calls only

## Article 8 — Separation of Concerns (Documentation)
- spec.md defines WHAT the system does — technology-agnostic
- plan.md / tasks.md defines HOW — all technical decisions live here
- bdd-scenarios.md defines acceptance criteria in plain English
- constitution.md defines constraints — referenced by all other documents

## Article 9 — Git Hygiene (Non-Negotiable)
- Nothing is ever committed directly to main. Ever.
- Each feature/phase is developed on its own branch
- Branch naming convention: feature/phase-N-description
  (e.g. feature/phase-2-direction-class)
- A feature branch is merged to main only when all three test
  layers pass (unit, integration, acceptance)
- Commit messages must be descriptive and reference the task
  (e.g. "feat: implement Direction class — Task 2.2")
- Each commit must leave the test suite green — no broken commits

## Article 10 — AI Agent Guardrails
- AI agent must not add dependencies not listed in Article 1
- AI agent must not implement wrapping behaviour
- AI agent must write tests before implementation (Article 2)
- AI agent must not generate functions exceeding Article 3 limits
- If the agent is uncertain about a requirement, it must flag it
  rather than assume

## Article 11 — Document Relationships

These documents must all be read before any implementation begins.
They form a dependency chain — each layer depends on the one above:

```
constitution.md  ← non-negotiable constraints. Overrides everything.
      ↓
spec.md          ← what the system does. Technology-agnostic.
      ↓
bdd-scenarios.md ← acceptance criteria. Derived from spec.md.
      ↓
tasks.md         ← how to build it. Every task must satisfy a scenario.
```

Resolution rules:
- If tasks.md conflicts with constitution.md, constitution wins
- If spec.md is ambiguous, check bdd-scenarios.md for the concrete example
- If a scenario has no corresponding task, flag it before proceeding
- Never proceed to a new phase without all tests in the current phase passing
- If any document is missing or unreadable, stop and report — do not assume
